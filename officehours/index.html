Tools for LaTeX to HTML conversion

A few projects in this space:

padding-bottom: 10em;

  • pandoc - Haskell, very versatile on input & output, extensible, large community
    limited math mode handling, barebones output

  • tex4ht/make4ht - Lua, available in texlive, good TeX input facilities, good fallback options
    Output tends to be layout focused, barebones output.

  • plasTeX - Python, focus on balance between structure and layout separation, good community, default styles
    Limited TeX input facilities, default output somewhat unstructured ("div-soup")

  • LaTeXML - Perl, available in texlive, good TeX input facilities
    Output very complex, output tends to be layout focused, math mode output limited

  • AMS texml / texml-to-html - Perl, good TeX input facilities, mathjax-oriented math mode handling, balanced output structure, good fallback options
    young project, AMS-oriented, output is barebones

Other interesting tools and projects

  • pretext - specialized XML language & tooling to create multiple outputs. A strong, mathematical community.
  • chirun - tools (partially using plasTeX) with a focus on author convenience
  • bookml - latexml-based tool to generate bookdown-oriented output
  • and many more

Making accessible content

Tools can never magically create accessible content.

Accessibility is a process focusing on people

  • The W3C Web Content Accessibility Guidelines (WCAG) is the legal basis in most countries. It lists ~60 "Success Criteria" across 3 Levels: A, AA, AAA.
    WCAG is very technical, specialist-oriented. Somewhat less formal introductions from the W3C are Introduction to Web Accessibility and Accessibility Principles. Many criteria revolve around applications and usually do not matter with long form content.
  • The WebAIM initiative provides many helpful articles

POUR Principles

WCAG's core principles: All content must be

  • Perceivable
  • Operable
  • Understandable
  • Robust

for all users.

Evaluating accessibility

Automated tests are not exhaustive but catch most frequent issues (~30% of WCAG is automatically testable but those cover ~50% of total errors, cf. Deque's research.

Popular general purpose tools: axe, WAVE.
Browsers also include tools such as Chrome Developer Tools (color picker, vision deficiency simulation, accessibility tree, accessibility audit)

Practical evaluation

Assistive technologies. A complex landscape of tools and devices. A key category is screenreaders:

  • Windows: NVDA, JAWS, Microsoft Narrator
  • Mac/iOS: VoiceOver
  • Android: Talkback
  • Linux: Gnome ORCA, Chromevox (ChromeOS only)

Other assistive technologies include zoom, dictation, and voice assistants software or devices.

More: Deque: Intro to Screen Readers (video), Demo: Sina's speed (video), SLCC Demo (video)

Understanding Screenreader interaction

Screenreaders are the primary tool for blind and low-vision users and commonly used by other users.

  • Content is linearized for reading and navigation.
    • users gather overview from structure not visual layout.
  • Reading is continuous or chunked (line, word, character)
  • Navigation facilitated by structural elements (landmarks, links, headings)
  • structural information is added to output (<h1>title<h1> voiced "title, heading level 1")
  • language markup content will influence output
  • screenreaders can pause, handle abbreviations, may skip punctuation and much more (see below)
  • Note. some screenreader users read as fast as visual users.

WebAIM resources: Designing for Screen Reader Compatibility, Testing with Screen Readers

A very quick and dirty list for mathematical long form content

  • Ensure "non-text" (e.g., an image) has a text alternative
  • Check use of color - contrast of text/background, do not use color alone, ensure contrasts in graphics etc.
  • Avoid images of text
  • Provide good document structure (landmarks, headings, enunciations)
  • For equation content, generate TeX for use with MathJax.
  • Avoid whitespace adjustments.
  • Properly link cross-references and provide good link text. Do not use sensory characteristics (e.g., "on the left", "the blue part")
